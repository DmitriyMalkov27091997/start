Руководство для разработчика программного обеспечения интеллектуальных средств автоматизации с поддержкой HART протокола.

 

Челябинск Апрель 2000

 

В первых реализациях HART устройств не рекомендуют программировать поддержку монопольного режима. Для сертификации его программировать необязательно. В настоящем руководстве программирование этого режима рассмотрено.

1. Передача данных между HART приборами

 

Обмен информацией между HART объектами выполняется в виде кадров. Кадр – это единица посылки данных пользователя по заданному адресу.

 

Кадры ограничены комбинациями символов, называемых преамбулами, уникальным символом «старт-кадра» (Delimiter), который определяет начало кадра, и полем Счетчика Байт, которое определяет конец кадра.

 

Источник и назначение данных внутри кадра определяется Адресным полем. Интерфейс между протоколом и пользователем осуществляется через поле Команда и поле Код Отклика, которое возвращает результат транзакции обратно к пользователю. Данные, если таковые существуют, передаются в поле Данных.

 

Все части кадра, включая ограничитель (Delimiter), должны быть защищены комбинацией контроля по четности в каждом передаваемом байте и завершающим полем Контрольного байта.

 

Все поля в кадре имеют целое число байт в длину и все байты кадра должны быть переданы в непрерывной последовательности, т.е. время между двумя соседними символами не должно быть больше времени передачи одного бита (1/1200 = 0.83 мс.) 1.1. Преамбула

 

Всем кадрам, передаваемым HART приборами, предшествует определенный набор шестнадцатеричных «FF» символов. Эти символы называются преамбулой кадра. Они обусловлены некоторыми задачами физического уровня для подготовки/синхронизации приемника.

 

Замечание: До разделителя (ограничителя) может быть получено несколько преамбул. Однако, две преамбулы, ближайшие к главному ограничителю, определяют начало кадра.

 

Для FSK физического уровня, HART приборы не должны передавать меньше 5 преамбул. Это дает время, достаточное для установления несущей частоты, и позволяет модему настроить его входные схемы для приема сообщения. Для HART приборов рекомендуется отправлять не более 20 преамбул.

 

Подчиненный прибор по умолчанию, должен передавать 5 преамбул. Если поддерживается Общая команда #59, главное устройство может устанавливать необходимое число преамбул, передаваемых подчиненным устройством. При идентификации Подчиненного прибора, (например с использованием Универсальной команды #0 или #11), Подчиненный прибор показывает минимальное число преамбул, которое должно передавать Главное устройство.

1.3. Адресация

 

Протокол использует адреса источника и приемника в каждом кадре (см. рис. 3.) И для длинного и для короткого адресов кадра старший бит представляет собой поле адреса мастера, связанного с этим кадром. Первичный мастер использует величину «1» для этого бита, вторичный мастер использует «0». Подчиненные приборы не должны в отклике менять это поле. Следующий бит - показатель монопольного режима подчиненного устройства. Подчиненные приборы должны установить этот бит в «1» для индикации того, что прибор находится в монопольном режиме и в «0», если нет. Главные устройства должны установить этот бит в «0» в запросах к подчиненным устройствам.


Протокол поддерживает кадры с длинными и короткими адресами. Какая форма адресов используется в сообщении, определяется ограничителем (см. рис.2).

 

Рис. 3 Адрес длинного кадра

 

1.3.1 Адреса длинного кадра

За исключением Команды 0, все HART кадры состоят из 5 байт адреса на основе уникального идентификатора (см. рис. 3). Уникальный идентификатор ассоциирован с каждым подчиненным устройством. Протокол обычно использует младшие 38 бит уникального идентификатора для адресации запросов канального уровня к подчиненным приборам. Как результат, адрес длинного кадра должен включать:

 

- Адрес мастера и бит монопольного режима, как описано выше в разделе 1.3 Адресация.

 

- Наименее значащие 6 бит – ID Производителя. Действующие ID Производителя - могут быть найдены в Таблице 8 : Идентификационные коды компаний производителей, спецификация “Общие таблицы” (Common Tables 8 HCF_SPEC-183)

ID производителя Метрана: 153 десятичное (99 в шестнадцатеричной системе исчисления).

Строка на индикаторе прибора: «Metran»

- 1 байт – Код Типа Прибора. Этот код назначается производителем. Поскольку только 6 младших битов ID Производителя используются в адресе, Код Типа Прибора должен быть назначен, как указано в табл. 1.1. Дополнительные спецификации относительно расположения Кодов Типа Прибора могут быть найдены в Краткой спецификации Команд (HCF Spec-99): Раздел 6.1 Правила совместимости;

 

- 3-х байтный Идентификатор Прибора. Это серийный номер, причем каждый прибор, сделанный с одинаковым Кодом Типа Устройства, должен иметь уникальный Идентификатор Прибора.

1.3.2 Кадры с короткими адресами


Предыдущие, устаревшие версии протокола использовали короткие адреса в кадрах. Для поддержки обратной совместимости с этими адресами универсальная команда #0 поддерживает кадры с короткой адресацией.

1.4. Команда

 

Байт команды содержит целое число (от 0 до FF), которое представляет собой код одной из команд HART. Если это код 254, то за байтом команды следует дополнительный байт, позволяющий кодировать более 256 команд. Протокол поддерживает один байт для команд (0-255) и два байта расширенных команд с номерами (256-65 535). Система команд делится на следующие классы:

 

Универсальные команды – набор команд, которые должны поддерживаться всеми HART совместимыми приборами. Все универсальные команды должны быть реализованы в соответствии с описанием универсальных команд (Universal Command Specification HCF_SPEC-127).

 

Общие команды – набор команд, применимых для широкого класса приборов. Общие команды должны поддерживаться приборами всегда, когда это возможно (см. Описание общих команд - Common Practice Command Specification HCF_SPEC-151).

 

Немножественные команды – специальные набор команд (от 122 до 126), разработанных только для использования в данной конструкции при производстве прибора .

 

Команды определенного класса приборов – набор команд, который позволяет установить и подготовить к эксплуатации полевые приборы без специальных драйверов этого устройства (см. Device Family Command Specification HCF_SPEC-160). Переменные Прибора классифицированы по типам процессов, которые поддерживаются этими приборами (давление, температура, расход и т.д.)

 

Специализированные команды – определяются производителем, согласно назначению полевого прибора.

1.5. Счетчик байт

 

Поле счетчика имеет длину один байт и содержит целое число, представляющее количество байт, которые формируют остальную часть сообщения (коды отклика и данные; байт контрольной суммы не включается в это количество). В этом поле допустимы числа от 0 до 255.

 

 

1.6. Поле данных

Этот раздел описывает содержание и организацию Поля Данных. В этом разделе рассмотрена группа полей, содержащих непосредственно измерительные данные, служебную информацию, связанную с передачей данных. Все эти поля зависят друг от друга и поэтому рассматриваются в комплексе. Поле данных поддерживает двунаправленный обмен между приборами в следующих случаях:

 

- Когда обмен успешен, Поле данных содержит  информацию об обмене между приборами.

- Для расширенных команд Поле данных включает два байта расширенного номера команды, позволяющих приборам однозначно определять информацию, содержащуюся в сообщении.

- Во всех откликах подчиненных устройств Поле данных содержит два байта Статуса Команды, обеспечивающие непрерывный обмен с главными приложениями.

- Когда в обмене с главным устройством обнаружена ошибка, отклик подчиненного устройства не содержит данных, определяемых номером команды. Это позволяет главному приложению очистить ошибочную индикацию.

Существует два формата запросов главного устройства (нормальная и расширенная команда) и восемь возможных форматов откликов подчиненных устройств.

Восемь форматов группы Полей данных состоят из различных комбинаций следующих шести подполей:

 


Рис. 6. Формат поля данных

 

- Байты данных запроса. Это подполе содержит данные для передачи от главного устройства к полевому прибору. Содержание и длина этого поля определяются в спецификации подаваемой командой. Для любой команды байты в этом поле всегда нумеруются от нуля, хотя первый байт в данных запроса может не быть первым байтом в Поле даных сообщения.

 

- Байты данных отклика. Это подполе содержит данные для передачи от полевого прибора к главному устройству. Содержание и длина этого поля определяются командной спецификацией. Байты этого поля всегда пронумерованы, начиная от нуля, хотя первый байт в данных отклика никогда не бывает первым байтом в поле данных сообщения.

 

- Расширенные командные числа. Это 16-битовый командный номер, используемый для расширения номера HART-команды до 65536. Команды 256-65536 могут не использоваться для специализированных команд (см..Device Family Command Спецификацию HCF_SPEC-160).

 

- Коммуникационный статус. Этот байт объединен с байтом кода отклика и показывает выявленные полевым прибором ошибки связи. Ошибка связи всегда показана в первом наиболее значащем бите этого байта. Когда полевой прибор не выявляет ошибок связи, код отклика на команду будет возвращен в сообщении отклика.

 

- Код отклика. Этот байт объединен с байтом коммуникационного статуса и показывает статус действия подчиненного устройства на запрос главного. Самый значимый бит (бит 7) кода отклика всегда нулевой.

 

- Статус прибора. Подполе статуса прибора обеспечивает высокий уровень иллюстрации состояния полевого прибора. Статус полевого прибора возвращается в каждом отклике подчинённого устройства для обеспечения непрерывной обратной связи между состоянием прибора и его функционированием.

 

Каждое сообщение не обязательно содержит все шесть подполей.

 

 

1.6.1 Код отклика

 

Код отклика включается только в ответное сообщение от подчиненного устройства. Он состоит из двух байт с информацией, закодированной поразрядно. В первом байте содержатся ошибки обмена данными, если они возникают. Если же обмен данными прошел без сбоев, в этом байте может содержатся статус полученной команды (например,  устройство занято или не удалось распознать команду). Во втором байте кода отклика содержится состояние функционирования подчиненного устройства.

При нормальной работе подчиненного устройства оба байта установлены в ноль.

Два байта кода отклика уникальны для кадров от Подчиненного устройства к Главному. Это поле содержит информацию, которая описывает результат выполнения транзакции:

- Если 7 бит первого байта отклика установлен – произошла коммуникационная ошибка и остальные 6 бит представляют краткое описание этой ошибки.

- Если 7 бит первого байта отклика очищен, оставшиеся биты несут информацию верхнего уровня протокола о выполнении или не выполнении команды.

- Второй байт кода отклика содержит информацию о статусе прибора и передаче данных. Он нулевой, если в первом байте содержится сообщение о коммуникационной ошибке.

Дополнительные рекомендации по использованию 2 байтов кода отклика могут быть найдены в HCF_SPEC_99 (Краткая спецификация на команды) и Приложении 1: Определения Кодов Отклика (HART-SMART Communications Protocol, Appendix 1 – Command Specific Response Code Definitions HCF_SPEC-307).

Оба бита должны быть установлены, когда какой-либо элемент конфигурации в полевом устройстве изменяется. Эти биты, использованные совместно со Счетчиком Изменений Конфигурации, позволяют главному провести мониторинг конфигурации полевого устройства.

Значение этого бита должно быть сохранено, даже если электропитание отключено от устройства или выполнена перезагрузка устройства.

 

Некоторые устройства позволяют делать диагностику, которая может прогнозировать требования для технического обслуживания. Когда полевое устройство имеет эту возможность, оно должно установить бит “Требуется обслуживание” для уведомления обслуживающего персонала.

 

Когда установлен бит “Доступен добавочный статус”, измерения корректны и подходяn для использования системами управления. Этот бит просто указывает, что команда #48 содержит диагностическую информацию, которая является полезной для главного устройства. Поскольку установка бита “Доступен добавочный статус” вызовет у большинства главных устройств выдачи Команды 48 и, как следствие этого, уменьшение полосы пропускания, проектировщики полевых устройств должны предусмотреть какая диагностика должна устанавливать этот бит.

 

Когда устройство находится в многоточечном (multi-drop) режиме или когда Токовая петля прямо (например, используя Команду #40) или косвенно (например, используя Команду #79) имитируется, должен быть установлен бит “Токовая петля фиксирована” (Loop Current Fixed). Обзор Канального Уровня

Канальный Уровень это уровень, где устанавливается структура HART протокола. HART - в основном протокол связи по схеме главный / подчиненный. Поддерживает два вида транзакций

 

· Первый тип транзакции – транзакция мастер/подчиненный при обычном режиме. В этом случае мастер инициализирует деятельность канала, посылая сообщение подчиненному. Соответствующий подчиненный посылает ответ, завершая этим транзакцию и освобождая канал.

· Второй тип транзакции - сообщение монопольного режима (МР). В этом случае подчиненное устройство МР выдает ответ на команду. Этот ответ - полная транзакция

 

 

Сообщения HART - одинаковы и для мастеров и для подчиненных. Сообщение состоит из разделителя, адреса, байтов расширения, команды, счетчика байта, данных и полей байта проверки (см. Рисунок 1). Каждое сообщение полностью определяет состояние уровня связи:

 

· Разделитель указывает тип фрейма; и

 

· Адрес указывает мастера и подчиненное устройство, которые поддерживают связь.

 

Рис. 1 HART Сообщение

 

До передачи сообщения посылается преамбула (см. Рисунок 2). Это необходимо для установления несущей и синхронизации модемов. Когда преамбула закончена, передается сообщение HART. Для FSK Физического уровня единичные байты передаются асинхронно и полная передача посылается без времени простоя между байтами.

 

Цель Уровня Связи - передача данных по HART каналу без ошибок. Для достижения этого в канальный уровень включены две основные функции:

 

· Кадрирование позволяет принимать сообщения. Другими словами, кадрирование позволяет идентифицировать начало и конец сообщения.

 

· Управление Доступом определяет, когда устройство может передать сообщение.

 

Рисунок 2 Передача Сообщения

 

Кадрирование

Синхронизация кадра или кадрирование - процесс идентификации и получения сообщения. Для HART Протокола, кадрирование начинается с конца преамбулы и приема разделителя. Для FSK Физического уровня начало сообщения определяется обнаружением двух байтов (0xFF's) преамбулы и разделителя, зафиксированного в последовательности без межсимвольного интервала. Четыре главных события происходят, которые приводят к успешному кадрированию сообщения:

 

•        Установление Несущей. Как только несущая установлена, канал считается активным и не-передающие устройства должны оставаться в этом состоянии(пассивном). Теория Связи показывает, что, когда корректная несущая присутствует в канале, другое устройство не сможет успешно передать сообщение.

 

•        Прием Разделителя. Это позволяет принять далее основную часть сообщения (адрес, байты расширения, команды и счетчика байт). Разделитель определяет структуру главной части сообщения:

 

-         Длинный или короткий формат кадра (определяется адресом), и

 

-         Присутствие любых байтов расширения фрейма.

 

Другими словами, разделитель позволяет измерить главную часть сообщения, которая должна быть вычислена.

 

•        Прием Счетчика Байт. Счетчик байт содержится в основной части сообщения и определяет число байт, которое формирует поле данных сообщения, именно это поле должно быть правильно создано.

 

•        Прием Байта Контрольной Суммы. Это указывает конец сообщения и является основным элементом в схеме проверки ошибок.

 

В то время как несколько ошибок могут быть обнаружены в течение приема сообщения, только три являются фатальными:ошибка задержки ; потеря несущей; и ошибка связи при приеме адреса или счетчика байт. Ошибка задержки указывает, что устройство передачи работало со сбоями при передаче середины сообщения. Потеря несущей указывает, что имеется сигнал недостаточной силы , чтобы можно было успешно связываться. Ошибка связи при получении адреса означает, что подчиненное устройство не знает приемника сообщения(было ли сообщение для него или нет),и, в результате, подчиненное устройство не будет отвечать. Ошибка в счетчике байта предотвращает успешное кадрирование сообщения. Все ситуации с этими ошибками фатальны, прерывают прием сообщения, и бракуют любое поле полученного сообщения. Управление Доступом к Среде

Управление доступом к среде HART (Media Access Control  (MAC)) разработано для того , чтобы обеспечить равный доступ к каналу для главного мастера, дополнительного мастера и для подчиненного устройства в активном режиме. Доступ к каналу управляется проходом маркера и использованием таймера выжидания другого мастера и таймера тихого канала (RT1 и RT2). Таймер тихого канала имеет различные продолжительности для двух мастеров. Прохождение маркера обозначено типом сообщения. Это означает что,  MAC зависит от идентификации:

 

•        Активности канала;

 

•        Тип фрейма, чтобы определить, кто следующим получит маркер; и

 

•        Конец сообщения, чтобы знать, когда инициализировать активность канала .

 

Разделитель - ключ к пониманию типа фрейма. HART имеет три фрейма :

 

•        STX - Это - сообщение от мастера подчиненному устройству. Маркер передается подчиненному устройству, адресованному в сообщении.

 

•        ACK - Это - отклик от подчиненного устройства на команду (STX) от мастера. Это завершает транзакцию главный / подчиненный и позволяет другому мастеру возможность связаться.

 

•        BACK - Это - сообщение от подчиненного устройства в активном режиме. Сообщение адресовано первичному или вторичному мастеру. В активном режиме ответ на обозначенный номер команды одинаков для обоих мастеров, каждый раз меняется лишь адрес главного устройства, попеременно устанавливается в 0 и 1. Маркер передается мастеру, не адресованному в сообщении подчиненного в активном режиме.

 

В некоторой документации символ O (например, OBACK, OSTX, OACK) используется, чтобы указать, что сообщение адресовано другому устройству HART.

 Восстановление Маркера

Наиболее серьезный запрос для протоколов перемещения маркера – это его восстановление, когда маркер потерян. Технические требования Канального Уровня обращаются к нескольким таймерам. Эти таймеры используются, чтобы обнаружить потерю маркера. Эти таймеры устанавливают временные границы доступа к каналу для прибора, у которого в данный момент находится маркер. 

 

Когда происходит передача маркера  (то есть в конце сообщения), таймер запускается. Приемник маркера должен начать свою транзакцию. Если начала следующей транзакции не наблюдается до переполнения таймера, то маркер считается потерянным. В таблице 2 показаны некоторые примеры процесса восстановления маркера.

 

Таймеры

Все таймеры выражены в символьных временах (9.167ms для Физического уровня КОДИРОВАНИЯ СО СДВИГОМ ЧАСТОТ). HART Протокол определяет следующие таймеры:

 

TTO  Таймер передачи(Таймер ожидания ответа от подчиненного). Время передачи 28 символов. Это - время, отведенное подчиненному устройству, чтобы оно начинало передавать ответ мастеру. 

 

HOLD         Таймер удержания маркера главными устройствами. Время передачи 2 Символов. Мастер с маркером (например, после последнего байта ACK-кадра, адресованного другому мастеру) должен начать свою передачу в пределах этого времени.

 

RT2    Таймер Предоставления Связи(Таймер предоставления связи другому мастеру). Время передачи 8 символов. Это - время, в течение которого данный мастер должен ожидать другого мастера перед тем, как начать транзакцию. Другими словами, другой мастер имеет маркер. Когда RT2 переполнится, маркер предполагается потерянным, и этот мастер может начинать свою следующую (другую) транзакцию (см. Последовательность 1)

 

BT      Таймер Активного Режима. Время передачи 8 символов. Это - время, в течение которого подчиненное устройство в активном режиме должно ожидать мастера, перед тем как начать транзакцию (см. Последовательность 2).

 

 

Когда мастер впервые включается в канал, он рассматривается как "не синхронизированный". Мастер может стать синхронизированным после переполнения RT1 таймера (в основном TTO + RT2) или,  идентификацией передачи сообщения в канале. RT1 имеет различные интервалы времени для первичного и вторичного мастеров.

 

RT1 (Для Первичного Мастера)     Таймер тихого канала или Таймер ожидания выхода в канал. Время передачи 33 символов. 

 

RT1 (Для Вторичного Мастера)     Время передачи 41 символа.

MAC Подчиненного

 

При понимании базовой структуры уровня доступа к среде может быть определен MAC уровень для Подчиненного. В нормальном режиме, подчиненное устройство пассивно и только отвечает на команды от мастера. Основная последовательность была бы:

 

1.       Подчиненное устройство неактивно (Простой).

         

2.       Несущая обнаружена, и сообщение принимается, адресованное подчиненному (Rx).

         

3.       Подчиненному устройству необходимо ответить в пределах TT0 (Tx Ожидаемый).

         

4.       Подчиненное устройство передает сообщение - ответ и возвращается в пассивное состояние (Tx).

 

Активный режим подчиненного устройства более сложен, потому что подчиненное устройство не надолго остается в пассивном состоянии, присоединяясь к каналу. Подчиненное устройство в активном режиме должно быть способно восстановить маркер, если другое подчиненное устройство будет не в состоянии отвечать на запрос мастера (Последовательность 4 в Таблице 2) и если мастер работает со сбоями, это необходимо, чтобы поддерживать связь (Последовательность 2). Анализ требований к MAC подчиненного приводит конечному алгоритму MAC подчиненного (Таблица 3 и Рисунок 4). Каждое из состояний обсуждается ниже.

 

Rx

Ключевое состояние - Rx, потому что полученное сообщение указывает состояние канала и кто получает маркер следующим. События, обязанные переходу из Rx - комбинация всех возможных (STX, ACK, и BACK) типов фрейма; адрес подчиненного в сообщении; и находится ли подчиненное устройство в активном режиме. Обнаружение Несущей вызывает переход в состояние Rx. Это не гарантирует, что сообщение получено (см. Рисунок 3). Так что возможно, что поиск несущей был шумовым переходным процессом или что была получена часть сообщения. Это приводит к 8 возможным комбинациям событий:

 

•        Нет сообщения или получена часть сообщения. Если это было вызвано шумовым переходным процессом или несколькими ложными символами, тогда данное событие должно игнорироваться. Если в результате данного события получена часть сообщения, тогда соответствующее действие неясно. Вероятнее всего, подчиненное устройство возвращается в пассивное состояние и ожидает истечение таймера, это лучший вариант.

 

•        Фатальная ошибка связи (в счетчике байта, или в поле адреса). Подчиненное устройство не знает кому было адресовано сообщение или размер и тип сообщения. Подчиненное устройство должно возвратиться в пассивное состояние и установить таймер активного режима, равным RT1 для восстановления  синхронизации канала.

 

•        "Хороший" STX, адресованный подчиненному устройству. В этом случае подчиненное устройство обязано реагировать(ответить). Подчиненное устройство переходит в состояние Tx Ожидание и устанавливает таймер передачи от подчиненного к главному, равным TT0.

 

•        STX, не адресованный данному подчиненному устройству ,которое находится в активном режиме. Данное подчиненное устройство должно быть готово восстановить маркер. Это делается установкой таймера BT, равным RT1, чтобы обеспечить максимальный интервал времени для другого подчиненного устройства, которое должно ответить на адресованный ему STX.

 

•        STX, не адресованный данному подчиненному устройству (в канале нет подчиненного в активном режиме). Подчиненное устройство не заботится относительно сообщения. Подчиненное возвращается в пассивное состояние.

 

•        ACK от другого подчиненного устройства, в то время как данное подчиненное устройство находится в активном режиме. Данное подчиненное устройство получает маркер и должно выпустить сообщение. Подчиненное устройство переходит в состояние Tx Ожидание и устанавливает(запускает) таймер передачи от подчиненного к главному TT0.

 

•        ACK от другого подчиненного устройства, в то время как нет устройства в активном режиме. Подчиненное устройство не заботится относительно сообщения. Подчиненное возвращается в пассивное состояние.

 

•        BACK от другого подчиненного устройства. Подчиненное устройство должно возвратиться к пассивному состоянию и сбрасывает BT до RT1.

 

 

Таблица 3 Матрица Изменения Состояния Подчиненного

 

Пассивное состояние (Простой)

Подчиненное устройство должно находится в Пассивном Состоянии ("не занято") большинство времени. Только два события вызывают переход в активное состояние:

 

•        Обнаружение Несущей. Это указывает, что имеется активность в канале, и подчиненное устройство должно пытаться получить(принять) сообщение. Это состояние не обязательно крайнее состояние. Например, подчиненное устройство могло реагировать на STX достаточно быстро, что несущая никогда не теряется.

 

•        Переполнения таймера активного режима. Таймер Активного режима (BT) только тогда активен, когда подчиненное устройство находится в активном режиме. Этот таймер равен RT2 при инициализации, т.е. при включении питания и поскольку маркер переходит, таймер может быть сброшен. Если этот таймер истекает,  подчиненное устройство должно восстановить маркер и выпустить сообщение подчиненного в активном режиме.

 

Запуск

На запуске подчиненное устройство не знает состояние в канале. Если имеется несущая, то сообщение может быть уже частично передано. Согласно Структурному анализу Подчиненное устройство предпринимает попытки захватить SOM (для КОДИРОВАНИЯ СО СДВИГОМ ЧАСТОТ(FSK) 2 0xFFs(преамбула) и разделитель) и далее принимает(получает) сообщение. Это плохая рекомендация со стороны Структурного анализа, потому что может иметься SOM, внедренный в поле данных (См. тесты канального уровня подчиненного устройства). Самый безопасный подход - это просто ждать несущую, чтобы уходить и переходить в пассивное состояние.

 

 

Tx Ожидание

Подчиненное устройство находится в состоянии Tx Ожидание каждый раз, когда это подчиненное устройство имеет маркер и должно разместить сообщение (и ACK или BACK) в канале. Это могло бы произойти, если BT истекает, STX, адресованный подчиненному устройству получен или если ACK (от этого подчиненного устройства или другой одного) находится на канале, и подчиненное устройство находится в активном режиме. Имеются три события, которые могут вызывать переход из состояния Tx Ожидание:

 

•        Ответ на передачу с прикладного уровня. В основном это означает, что прикладной уровень подчиненного устройства обработал команду и имеет ответ для мастера. Подчиненный переходит в состояние Tx

 

•        Отклик подчиненного в активном режиме с прикладного уровня. Прикладной уровень запущен примитивом (базисной функцией) канального уровня cyclic_update.request и реагирует(отвечает) cyclic_update.confirm, чтобы позволить подчиненному устройству перейти в состояние Tx.

 

•        Передача по истечение срока(TTO). Прикладной уровень не смог ответить в пределах TT0. Это - плохая вещь для подчиненного устройства делать таким образом, ответ после того, как TT0 переполняется(истекает), может вызывать коллизии в канале. Подчиненное устройство должно перейти в пассивное состояние.

 

Tx

Это состояние вызывает передачу сообщения. Имеются три события:

 

•        Передан ACK, в то время как подчиненное устройство не находится  в активном режиме. В этом случае подчиненное устройство может перейти к Простою(пассивное состояние) (оно не имеет никаких других обязанностей)

 

•        Передан ACK, в то время как данное подчиненное устройство находится в активном режиме. Если подчиненное устройство передает ACK, в то время как находится в активном режиме, то оно должно немедленно выпустить сообщение от себя(burst). Подчиненное устройство должно перейти назад к состоянию Tx Ожидание, чтобы позволить прикладному уровню выпустить сообщение (burst).

 

•        Передан BACK. Это подразумевает, что подчиненное устройство находится в активном режиме. После BACK подчиненное устройство должно быть подготовлено, чтобы восстановить маркер, если мастер не выпускает STX. Это делается установкой BT, равным RT2.

 

Не должно быть никаких ошибок в передаче.

 

Рисунок 4 Диаграмма переходов Подчиненного

 

MAC Главного

Понимание маркера, подразумеваемого в соответствии с каждым сообщением, позволяет разработать конечный автомат для мастера. В нормальном режиме главное устройство инициализирует HART связь. Основная последовательность:

 

1.       Мастер соединяется с каналом.

 

2.       После истечения таймера RT1(таймер тихого канала) мастер становится синхронизированным с каналом.

 

3.       После этого синхронизированный мастер должен начать передачу в течении времени, определенного таймером HOLD(таймер удержания маркера мастерами).

 

4.       Как только сообщение послано, подчиненному устройству требуется начать его отклик в пределах TTO.

 

5.       Подчиненное устройство передает ответ и мастер запускает таймер RT2(таймер предоставления связи другому мастеру).

 

6.       Если другого мастера нет, то таймер Предоставления Связи(RT2) истекает.

 

7.       Мастер должен начать другую передачу (то есть, переход на 3 пункт)

 

Мастер несет больше всех ответственности за восстановление при наличии ошибок. Например, если другого мастера нет (Последовательность 1 в Таблице 2) или если подчиненное устройство работает со сбоями при ответе ( Последовательность 3), то это обязанность мастера восстановить маркер. Более того, в то время как подчиненное устройство может игнорировать частичное сообщение или то, которое не может быть создано, мастеру необходимо опять восстановить синхронизацию с каналом перед посылкой другого сообщения. 

 

Анализ MAC требований для главного устройства приводит к конечному автомату для MAC главного устройства (Таблица 4 и Рисунок 5). Каждое из состояний обсуждается ниже.

 

Rx

Как и для MAC подчиненного, ключевое состояние - Rx, потому что полученное сообщение указывает состояние канала и того, кто получит маркер следующим. События, которые приводят к перемещению из Rx состояния, - это тип полученного фрейма; адрес мастера в сообщении; и имеется ли в канале подчиненное устройство в активном режиме. Из этого следует 9 возможных комбинаций событий:

 

•        Нет сообщения или частично полученное сообщение. Если это было вызвано шумовым переходным процессом или несколькими ложными(поддельными) символами, тогда это явление должно игнорироваться. Если в результате данного события получена часть сообщения, тогда следующее действие, соответствующее этому событию, неясно. Вероятно, просто возвращение к простою и ожидание истечения таймера, это лучший вариант.

 

•        Фатальная ошибка связи (в счетчике байта, или поле адреса). Это указывает, что имелось сообщение на канале. Однако сообщение не могло быть получено (кадрировано). Мастер не знает, кто имеет маркер, кому было адресовано сообщение или размер и тип сообщения. Мастер должен возвратиться к Простою и установить таймер RT1(таймер тихого канала), равным RT1, чтобы позволить каналу восстановить синхронизацию.

 

•        STX от другого мастера. Подчиненное устройство должно ответить на это сообщение. Однако, мастер должен быть подготовлен, чтобы восстановить маркер, если подчиненное устройство не отвечает. Таймер тихого канала(RT1) устанавливается равным RT1 и мастер возвращается к Простою.

 

Таблица 4 Матрица Изменения Состояния Мастером

 

•        STX от этого мастера. Это - серьезная ошибка. HART Протокол является полудуплексным и модемы не могут получать сообщение, в то время как они передают. Единственный случай, когда это может происходить , если другой мастер, находящийся в канале, имеет такой же адрес. Это требует вмешательства пользователя.

 

•        BACK этому мастеру. Маркер переходит к другому мастеру. Этот мастер должен быть подготовлен, чтобы восстановить маркер. Это делается установкой таймера RT1(таймера тихого канала),равным RT1+RT2????, чтобы предоставить максимальное время другому мастеру, чтобы он начал транзакцию.

 

•        ACK другому мастеру, в то время как канал находится в активном режиме. Маркер переходит к подчиненному устройству, которое находится в активном режиме. Мастер должен установить таймер RT1(таймер тихого канала), равным RT1, и возвратится к Простою, чтобы ожидать сообщение от подчиненного устройства, которое находится в активном режиме(burst).

 

•        ACK этому мастеру, в то время как канал находится в обычном режиме. Другой мастер получает маркер. Этот мастер устанавливает таймер предоставления связи(RT2), равным RT2, и переключается в состояние Простой.

 

•        BACK другому мастеру. Этот мастер получает маркер. Мастер переключается в состояние Tx Ожидание и должен начать передачу в пределах времени удержания маркера мастерами(HOLD).

 

•        ACK другому мастеру, в то время как канал находится не в активном режиме. Этот мастер имеет маркер (нет никакого подчиненного устройства в активном режиме, которое надо было бы ждать). Мастер переключается в состояние TX Ожидание и должен начать передачу в пределах времени удержания маркера мастерами (HOLD).

 

Простой

Мастер обычно возвращается к простою, чтобы ждать другое сообщение или истечение таймера(время ожидания). Возможны три события:

 

•        Обнаружение Несущей. Это указывает, что имеется активность в канале, и подчиненное устройство должно предпринимать попытки получить сообщение. Это явление - не обязательно крайнее состояние. Например, подчиненное устройство могло реагировать на STX достаточно быстро, так  что несущая никогда не теряется.

 

•        Истечение таймера RT1 (таймер тихого канала). Когда это происходит, мастер знает, что канал был пуст (тих) достаточно долго, чтобы стать синхронизированным, или, если подчиненное устройство не смогло ответить на запрос. Мастер переключается в состояние TX Ожидание, теперь может создать маркер и передавать сообщение. 

 

•        Истечение таймера RT2 (Таймера Предоставления Связи Другому Мастеру). Таймер предоставления связи используется, чтобы обеспечить другому главному устройству доступ к каналу. Если этот таймер истекает, тогда вероятнее всего другого мастера нет. Мастер переходит в состояние TX Ожидание, создает маркер и передает сообщение.

 

Запуск

При запуске мастер не знает состояние о канале. Если имеется несущая, тогда сообщение может быть уже частично передано. Мастер остается в этом состоянии, пока не произойдет потеря несущей.

 

Tx Ожидание

Мастер находится в состоянии Tx Ожидание всякий раз, когда мастер имеет маркер и должен разместить сообщение (STX) на канале. Это могло бы произойти, если таймер истекает или если маркер был передан мастеру. Имеются два события, которые могут вызывать переход из состояния Tx Ожидание:

 

•        Запрос на передачу от прикладного уровня. В основном это означает, что прикладной уровень подчиненного устройства обработал команду и имеет ответ для мастера. Подчиненное устройство переходит в состояние Tx

 

•        Истечение таймера HOLD(таймер удержания маркера мастерами). Прикладной уровень оказался не в состоянии сгенерировать запрос прежде, чем таймер HOLD истекает. Мастер должен восстановить синхронизацию к каналу,  устанавливая таймер тихого канала(RT1), равным 2*RT1

 

Tx

Это состояние вызывает передачу сообщения. Мастер остается в этом состоянии так долго, сколько необходимо, чтобы послать сообщение:

 

•        Передача Закончена. Сообщение было послано подчиненному устройству. Мастер устанавливает таймер тихого канала, равным RT1, чтобы дать время подчиненному устройству на ответ, и возвращается к Простою.

 Коды отклика (HCF_SPEC-307)

1. Резюме

 

Данное приложение содержит коды отклика (статус полученной команды) на команды, специфичные для конкретного датчика. Коды отклика находятся в первом байте двухбайтного поля отклика (битах #0-#6) в случае отсутствия ошибок передачи данных. Коды отклика – это 7-битные целые положительные числа. в данном приложении приводится не побитовое отображение кодов.

Коды отклика делятся на два класса: коды- ошибки и коды- предупреждение. Коды- ошибки указывают, что во время выполнения команды произошла ошибка и выполнение команды не будет завершено. В таком случае байты данных ответа не возвращаются. Коды- предупреждения начинаются с сообщения “Предупреждение”. Они указывают, что команда была принята и успешно выполнена, но определенное состояние, в котором находится датчик, влияет на результаты. Байты данных ответа возвращаются как данные, измененные датчиком.

Некоторые коды отклика применимы для любой транзакции(коды отклика с одним значением), другие коды специфичны для конкретных команд, то есть эти коды имеют различное значение для различных команд(коды отклика с несколькими значениями). В таблице 1 приведены характеристики кодов отклика.

Создание упакованных ASCII-символов:

Создание строки ASCII-символов- это отбрасывание 2-х старших битов от каждого символа и сжатие результата:

1. Отбросить бит #6 и #7 от каждого символа.

2. Упаковывание 4-х 6-битных ASCII-символа в 3 байта.

3. Повторить, пока не будет получена целая строка.

Этот алгоритм может быть выполнен маскированием и сдвигом 4-х 6 битовых в 24 битовый регистр с последующим преобразованием трех байт в упакованную ASCII стоку.

 

Восстановление ASCII-символов:

Для восстановления строки ASCII-символов в несжатом варианте необходимо:

1. Распакуйте 4 6-ти разрядных ASCII символа.

2. Для каждого символа поместите содержимое бита 5 в 6 бит.

3. Для каждого символа установите бит7 равным 0

4. Повторите, пока не будет получена целая строка.

Этот алгоритм может быть реализован загрузкой трех байт в 24 битовый регистр и преобразованием 6 битовых символов в строку. Проверьте полученный символ преобразованием в 6 бит если необходимо.

 

 

Представление даты

 

Дата представляется тремя 8 – битовыми двоичными беззнаковыми целыми числами, описывающими соответственно день, месяц и год (плюс 1900). Дата передается в следующей последовательности: день-байт, месяц-байт и год-байт. Это позволяет представить любую дату между 1 января 1900 и 31 декабря 2155.

 

 

Форматы с плавающей точкой

 

IEEE-754 поддерживает форматы с плавающей точкой одинарной и двойной точности. Числа с плавающей точкой состоят из 3 частей: знак(1 бит), экспонента и мантисса.

S       знак мантиссы (1=”отрицательное число”)

E       экспонента

F       мантисса, которая всегда между 1,0 и 2,0, целая часть мантиссы всегда 1. Целая часть мантиссы не включается в стандарт одинарной точности IEEE-754и двойной.

Бесконечно большие числа(NaN)

 

NaN показывает числа, которые не могут быть интерпретированы. Стандарт IEEE разрешает данные числа со знаком или без знака. NaN со знаком не может быть получены в результате вычислений. Таким образом, NaN со знаком и без знака (0х7F, 0хA0, 0х00, 0х00) разрешаются в некоторых спецификациях на команды, чтобы указывать, когда полевое устройство не поддерживает значения определённых данных.
